MODULE_03_STEP_1 — DIRECTORY DATA STRUCTURE
Цель шага

Создать новый, чистый data-слой DIRECTORY, который:

не зависит от старых карточек
соответствует новому генплану
может спокойно использоваться и ботом, и будущим сайтом
станет единственной базой для дальнейшего loader/UI
Результат шага

После STEP 1 у вас должно быть:

папка хранения объектов DIRECTORY
единый стандарт карточки
1 эталонная карточка нового формата
понимание, что старые карточки больше не считаются стандартом
1. Целевая структура папок

Новый слой:

app/data/
 ├ city_events/
 ├ objects/
 ├ sources/

Для DIRECTORY сейчас работает именно:

app/data/objects/

Внутри на текущем этапе:

app/data/objects/
 ├ BEAUTY_001.json

Пока без index.json, без усложнения.
Сначала — отдельные карточки как отдельные сущности.

2. Правило хранения
Основное правило

1 карточка = 1 JSON-файл

Это даёт:

простую проверку
простую миграцию
простое редактирование
меньше риска всё сломать одним файлом
Именование файлов

Имя файла = ID карточки

Пример:

BEAUTY_001.json
TRANSLATION_001.json
TRANSPORT_001.json
Формат ID

Рекомендуемый стандарт:

CATEGORYCODE_XXX

Примеры:

BEAUTY_001
TRANSLATION_001
TRANSPORT_001

Не надо сейчас делать сложные slug-имена.

3. Стандарт карточки

На STEP 1 берём минимальный обязательный стандарт.

Обязательные поля
{
  "id": "BEAUTY_001",
  "category": "beauty",
  "title": "Olesya Hair",
  "description_short": "Женский парикмахер в Алании.",
  "contact": "@olesya.hair07",
  "request_enabled": true,
  "updated_at": "2026-03-21"
}
Значение полей
id — уникальный ID карточки
category — категория строго из разрешённого списка
title — название карточки
description_short — короткое понятное описание
contact — основной контакт
request_enabled — можно ли отправлять заявку
updated_at — дата последнего обновления
Что пока НЕ обязательно на STEP 1

Пока не усложняем. Не обязательно добавлять:

description_full
phone
location
image
partner_chat_id
partner_status

Это можно расширить на следующем шаге или отдельной ревизией.
Сейчас важнее собрать живой стандарт, а не “идеальную карточку на 30 полей”.

4. Разрешённые категории

На этом шаге использовать только категории из MASTER PLAN:

beauty
transport
translation
home_services
sport
tourism
real_estate
health
daily

Если старая карточка не попадает в этот список напрямую — подгоняем только через одну из этих категорий, без самодеятельности.

5. Статус старых карточек

Важно зафиксировать жёстко:

старые карточки не используются как стандарт

Они допускаются только как:

источник имени
источник описания
источник контакта
источник данных партнёра

То есть:

❌ не “чиним старую карточку”
✔️ создаём новую карточку по новому формату и переносим в неё данные вручную

6. Эталонная карточка для старта

Для STEP 1 достаточно одной карточки.

И логично взять вашу beauty-карточку как сырьё и собрать из неё новый чистый файл.

Пример эталона:

{
  "id": "BEAUTY_001",
  "category": "beauty",
  "title": "Olesya Hair",
  "description_short": "Женский парикмахер в Алании.",
  "contact": "@olesya.hair07",
  "request_enabled": true,
  "updated_at": "2026-03-21"
}

Если контакт должен быть другим — подставляете реальный основной контакт.

7. Что нужно сделать физически
Создать папку:
app/data/objects/
Создать файл:
app/data/objects/BEAUTY_001.json
Вставить в него новую карточку по стандарту
8. Критерий готовности STEP 1

STEP 1 считается выполненным, если:

папка app/data/objects/ создана
в ней есть хотя бы 1 новая карточка
карточка сохранена как отдельный .json
карточка содержит все обязательные поля
category валидна
карточка не опирается на старую структуру
9. Что НЕ делаем на этом шаге

Сейчас не делаем:

loader
category menu
object list
open card handler
partner request flow
index file
картинки
много карточек сразу

Сейчас только структура и эталон.
