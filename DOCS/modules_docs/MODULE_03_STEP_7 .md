MODULE_03_STEP_7 — POLISH + INTEGRATION
1. Назначение

Этот шаг отвечает за:

сшивку всех частей DIRECTORY в один живой поток

После него DIRECTORY должен работать как цельный модуль, а не как набор отдельных кусков.

2. Что именно интегрируется

К этому шагу уже существуют логически:

CATEGORY MENU
→ OBJECT LIST
→ CARD VIEW
→ PARTNER REQUEST FLOW

STEP 7 нужен, чтобы проверить и закрепить:

переходы

возвраты

согласованность текстов

честные fallback

отсутствие тупиков

3. Полный маршрут модуля

Финальный маршрут DIRECTORY:

MAIN MENU
→ Услуги
→ CATEGORY MENU
→ OBJECT LIST
→ CARD VIEW
→ REQUEST FLOW
→ CARD VIEW / FINAL
4. Главная цель polish
ПОЛЬЗОВАТЕЛЬ ДОЛЖЕН ПРОЙТИ ПУТЬ БЕЗ ПУТАНИЦЫ
5. Что проверяем по UX
5.1 CATEGORY MENU

Проверяем:

показываются только живые категории

нет пустых категорий

есть back в главное меню

текст короткий и понятный

5.2 OBJECT LIST

Проверяем:

список не перегружен

кнопки карточек читаемы

back возвращает к категориям

пустая категория не ломает экран

5.3 CARD VIEW

Проверяем:

нет пустых блоков

контакты показываются аккуратно

услуги читаются списком

есть действия

есть back в список

5.4 REQUEST FLOW

Проверяем:

понятные подсказки

подтверждение перед отправкой

отмена работает

финал не тупиковый

6. Интеграционные правила
6.1 Один маршрут назад на каждом уровне

Возвраты должны быть строго предсказуемыми:

REQUEST → CARD
CARD → LIST
LIST → CATEGORY
CATEGORY → MAIN MENU

Не должно быть хаотичных прыжков.

6.2 Единый стиль текста

Во всём DIRECTORY:

короткие фразы

без рекламного шума

без разных тонов на каждом экране

6.3 Единый стиль кнопок

Не должно быть такого:

где-то “Назад”

где-то “Вернуться”

где-то “Обратно”

Нужно выбрать один стиль и держать его.

6.4 Единые callback-префиксы

Логика callback должна быть стабильной.

Примерно так:

dir_menu
dir_cat:<slug>
dir_obj:<id>
dir_req:<id>

Без хаотичных названий.

7. Что исправляется на step 7

Это не шаг новых функций.
Это шаг доводки.

Допустимо:

сократить текст

убрать дубли кнопок

почистить названия

выровнять back-логику

убрать лишние поля из карточки

поправить пустые состояния

Недопустимо:

изобретать новый функционал

менять модель карточки

ломать маршрут

8. Пустые состояния

Все пустые состояния должны быть честными и короткими.

Если нет категорий:
Сейчас раздел ещё наполняется.
Если нет объектов:
Сейчас в этой категории нет доступных карточек.
Если карточка не найдена:
Карточка не найдена.
Если нет заявок:
Партнёр пока не подключил приём заявок.
9. Ошибки и устойчивость

DIRECTORY не должен падать, если:

одна карточка битая

категория пустая

partner_chat_id отсутствует

нет фото

нет одного из контактов

Модуль должен:

пропускать битые данные на loader уровне

честно отвечать в UI

не валить поток пользователя

10. Связка с MODULE_01

DIRECTORY обязан встроиться в CORE NAVIGATION.

То есть:

MAIN MENU → Услуги → DIRECTORY

И обратно:

DIRECTORY → MAIN MENU

Без отдельной “вселенной” DIRECTORY.

11. Связка с MODULE_02

DIRECTORY использует только:

app/data/objects/

Никаких:

карточек в коде

карточек в UI

карточек в нескольких папках

12. Минимальный интеграционный smoke test

Перед lock модуль должен пройти вот такой путь:

1. Открыть "Услуги"
2. Увидеть категории
3. Открыть категорию
4. Увидеть список карточек
5. Открыть карточку
6. Нажать "Оставить запрос"
7. Пройти flow
8. Вернуться назад
9. Вернуться в меню

Если этот путь проходит без ошибок — модуль живой.

13. Invariants step 7
1. весь DIRECTORY работает как один маршрут
2. все возвраты предсказуемы
3. UI не перегружен
4. callback-логика едина
5. данные приходят только из loader
6. пустые состояния честные
7. ошибки не валят модуль
14. Ready check

STEP 7 считается готовым, если:

1. CATEGORY MENU работает
2. OBJECT LIST работает
3. CARD VIEW работает
4. REQUEST FLOW работает
5. все back-переходы работают
6. нет тупиков
7. нет падений на пустых данных
8. DIRECTORY встроен в MAIN MENU
9. тексты и кнопки выровнены
15. Финальный смысл step 7

После этого шага DIRECTORY уже можно считать не “планом”, а реальным рабочим модулем:

пользователь → находит услугу → открывает карточку → отправляет запрос
