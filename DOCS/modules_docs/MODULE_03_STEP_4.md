MODULE_03_STEP_4 — DIRECTORY OBJECT LIST
1. Назначение

Этот шаг отвечает за:

CATEGORY → список карточек внутри выбранной категории

То есть пользователь уже выбрал категорию, и теперь система показывает ему реальные активные объекты этой категории.

2. Место в маршруте
MAIN MENU
→ Услуги
→ CATEGORY MENU
→ OBJECT LIST
→ CARD

На этом шаге реализуем:

CATEGORY MENU → OBJECT LIST
3. Источник данных

OBJECT LIST строится только через loader.

Источник:

loader.get_objects_by_category(category)

Это важно.

UI не должен:

читать JSON напрямую

сам фильтровать файлы

вручную собирать список

4. Главное правило
В СПИСОК ПОПАДАЮТ ТОЛЬКО ACTIVE КАРТОЧКИ ВЫБРАННОЙ КАТЕГОРИИ
5. Формат экрана

Экран должен быть коротким и понятным.

Пример:

💇 Красота

👇 Это список. Нажмите на название, чтобы открыть карточку.

Дальше идут кнопки карточек.

6. Что показывается в списке

На уровне списка не нужно перегружать.

Каждый элемент списка должен показывать:

название

при необходимости очень короткое уточнение

Базово достаточно:

Olesya Hairstylist

Можно позже добавить 1 короткую строку, но на baseline лучше не раздувать.

7. Формат кнопок объектов

Одна карточка = одна кнопка.

Callback формат:

dir_obj:<card_id>

Примеры:

dir_obj:beauty_olesya_001
dir_obj:transport_alanya_001
8. Кнопка назад

Из OBJECT LIST пользователь должен вернуться в:

CATEGORY MENU

Обязательная кнопка:

⬅️ Назад
9. Пустая категория

Теоретически такого быть почти не должно, потому что CATEGORY MENU показывает только непустые категории.

Но защита нужна.

Если категория открылась, а активных карточек нет:

Сейчас в этой категории нет доступных карточек.

И кнопка назад.

10. Сортировка списка

Список должен быть стабильным.

Порядок:

sort_order → name

То есть:

сначала системный порядок

потом имя как резервный порядок

11. Что НЕ делаем на этом шаге

❌ не показываем полный текст карточки
❌ не выводим все контакты прямо в списке
❌ не вставляем фото
❌ не делаем большие описания
❌ не смешиваем список и карточку в одном экране

12. UI логика

OBJECT LIST должен:

получить category

запросить список у loader

отрендерить текст

построить кнопки карточек

дать кнопку назад

И всё.

13. Нужные части модуля

Логически понадобится:

directory/
 ├ ui/
 │   ├ handlers.py
 │   ├ callbacks.py
 ├ render/
 │   ├ object_list_render.py

Минимум:

handler открытия списка категории

callback parser для dir_obj:<id>

render текста списка

builder клавиатуры списка

14. Инварианты step 4
1. список строится только через loader
2. в список попадают только active карточки
3. каждая кнопка ведёт в CARD
4. есть BACK к категориям
5. UI не падает на пустой категории
6. список не перегружен
15. Ready check для step 4

Шаг считается готовым, если:

1. категория открывает свой список
2. в списке только active карточки
3. кнопки карточек работают
4. back возвращает к категориям
5. пустая категория даёт честный ответ
6. сортировка стабильна
16. Edge case: битая карточка

Если loader уже пропустил невалидную карточку, UI о ней ничего не знает.

Это правильно.

OBJECT LIST не должен заниматься лечением битых данных.

17. Смысл этого шага

Это первый момент, когда DIRECTORY начинает выглядеть как живая система:

категория → реальные исполнители

Без этого у вас есть только структура, но нет движения.
