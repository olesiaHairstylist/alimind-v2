MODULE_03_STEP_2 — OBJECTS LOADER SPEC
1. Назначение

Загрузчик объектов отвечает за:

чтение карточек из app/data/objects/
→ проверку базовой валидности
→ отбор активных карточек
→ подготовку данных для UI

Он не рендерит UI и не решает навигацию.
Он только даёт системе чистый список объектов.

2. Источник

Только одна папка:

app/data/objects/

Правило:

1 файл = 1 карточка

Читаются только:

*.json
3. Что должен делать loader
3.1 Сканировать папку

Найти все JSON-файлы в app/data/objects/.

3.2 Читать файлы

Открыть каждый файл и загрузить JSON.

3.3 Проверять базовую валидность

Минимум:

JSON должен быть объектом

обязательные поля должны существовать

status должен быть допустимым

category должна быть допустимой

services должен быть списком

sort_order должен быть числом

3.4 Отфильтровать неактивные

В UI попадают только:

status = active
3.5 Отсортировать

Сначала по:

category

потом по:

sort_order

потом по:

name
3.6 Отдать готовые структуры

Чтобы UI не лазил по сырым JSON.

4. Что loader НЕ должен делать

❌ не должен:

читать карточки из кода

знать про Telegram кнопки

отправлять сообщения

менять JSON

автоматически “чинить” битые карточки

решать логику заявок

Если карточка битая — loader её пропускает или логирует, но не превращает в магию.

5. Выходные сущности

На старте достаточно двух представлений.

5.1 Полный объект карточки

Для открытия карточки.

5.2 Короткий список для UI

Для списка внутри категории.

6. Рекомендуемая внутренняя модель

Логически loader должен уметь вернуть:

A. Все активные карточки
load_all_active_objects() -> list[ObjectCard]
B. Все категории, в которых есть активные карточки
get_active_categories() -> list[str]
C. Все активные карточки одной категории
get_objects_by_category(category: str) -> list[ObjectCard]
D. Одну карточку по id
get_object_by_id(card_id: str) -> ObjectCard | None
7. Правила валидации baseline

Карточка считается пригодной для загрузки, если есть:

id
status
is_partner
category
name
title
short_description
services
location
image
request_enabled
updated_at
sort_order
Дополнительно:

status ∈ {active, inactive}

is_partner = true

category из разрешённого списка

services не пустой список

image строка

sort_order число

Условная проверка:

Если request_enabled = true, то должен быть:

partner_chat_id
Контактная проверка:

Хотя бы одно поле не пустое:

phone / telegram / whatsapp / instagram / site_url
8. Политика ошибок

Если JSON сломан:

не падать всем модулем

логировать

пропустить файл

Если карточка невалидна:

логировать

пропустить файл

Если папка пустая:

вернуть пустой список

UI потом честно показывает:

Сейчас нет данных.
9. Структура файлов модуля

Логично сделать так:

app/modules/directory/
 ├ contracts/
 ├ loader.py
 ├ validator.py
 ├ models.py
 ├ ui/
 └ render/

Минимум для step 2:

models.py
validator.py
loader.py
10. models.py

Здесь описывается структура карточки в коде.

Примерно логически:

ObjectCard

возможно позже ObjectCardSummary

Но сейчас достаточно одной полной модели.

11. validator.py

Отвечает за:

проверку обязательных полей

проверку значений

решение: валидна карточка или нет

Возвращает:

True/False
или

список ошибок

На старте можно просто True/False + print/log.

12. loader.py

Отвечает за:

поиск файлов

чтение JSON

прогон через validator

сбор активных карточек

сортировку

выдачу функций доступа

13. Инварианты step 2
1. loader читает только app/data/objects/
2. loader не зависит от UI
3. невалидные карточки не ломают систему
4. в UI попадают только active карточки
5. источник объектов только один
6. карточки не хранятся в коде
14. Ready check для step 2

Step считается готовым, если:

1. loader читает все JSON из app/data/objects/
2. невалидные файлы не валят систему
3. active карточки возвращаются списком
4. можно получить карточки по category
5. можно получить карточку по id
6. сортировка стабильная
