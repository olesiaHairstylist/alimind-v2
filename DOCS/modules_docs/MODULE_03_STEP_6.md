MODULE_03_STEP_6 — PARTNER REQUEST FLOW
1. Назначение

Этот шаг отвечает за:

CARD VIEW → отправка запроса партнёру

Это ключевой бизнес-шаг DIRECTORY.
Именно здесь карточка перестаёт быть просто страницей и становится рабочей точкой связи.

2. Место в маршруте
MAIN MENU
→ Услуги
→ CATEGORY MENU
→ OBJECT LIST
→ CARD VIEW
→ REQUEST FLOW
3. Главный принцип
ЗАЯВКА = КОРОТКИЙ, ПОНЯТНЫЙ, УПРАВЛЯЕМЫЙ ПОТОК

Не анкета на 20 полей.
Не CRM.
Не “заполните всё о себе”.

4. Когда доступен flow

Запрос доступен только если у карточки:

request_enabled = true
partner_chat_id != empty

Если нет — честное сообщение:

Партнёр пока не подключил приём заявок.
5. Структура потока

Минимальный baseline:

STEP 1 — сообщение
STEP 2 — контакт
STEP 3 — подтверждение
STEP 4 — отправка партнёру
STEP 5 — финал
6. STEP 1 — сообщение пользователя

После нажатия:

📨 Оставить запрос

Пользователь получает короткую инструкцию:

Напишите, что вам нужно.

Пример:

нужна запись на стрижку

нужен трансфер в аэропорт

нужен переводчик на завтра

Требование

Сообщение должно быть:

обязательным

не пустым

без лишних ограничений

7. STEP 2 — контакт пользователя

После текста запроса:

Оставьте контакт для связи.

Это может быть:

телефон

Telegram

WhatsApp

любой удобный способ

Требование

Контакт обязателен.

8. STEP 3 — подтверждение

Перед отправкой показывается короткое резюме:

Ваш запрос:
<текст>

Ваш контакт:
<контакт>

Отправить партнёру?

Кнопки:

✅ Отправить
✏️ Изменить
❌ Отмена
9. STEP 4 — отправка партнёру

После подтверждения бот отправляет сообщение в partner_chat_id.

Формат сообщения партнёру

Базовый вариант:

📨 Новый запрос из AliMind

Категория: <category>
Карточка: <name>

Запрос:
<текст пользователя>

Контакт:
<контакт пользователя>

Можно добавить:

Источник: AliMind Directory
10. STEP 5 — финал для пользователя

После успешной отправки:

Запрос отправлен.
Партнёр получит его в Telegram.

И дальше:

кнопка назад
или

кнопка в меню

11. Что должно передаваться в flow

Минимально:

card_id
partner_chat_id
category
name

Это нужно, чтобы:

понимать, из какой карточки пришёл запрос

правильно подписывать сообщение партнёру

12. FSM логика

Нужны состояния:

request_message
request_contact
request_confirm

Этого достаточно.

Не надо:

10 состояний

ветвления на полпроекта

дополнительных опросников

13. Кнопки и поведение
В карточке:
📨 Оставить запрос
На подтверждении:
✅ Отправить
✏️ Изменить
❌ Отмена
14. Обработка отмены

Если пользователь отменил:

Запрос отменён.

И возврат:

либо в карточку

либо в список

Лучше в карточку, чтобы не терять контекст.

15. Edge cases
15.1 Нет partner_chat_id

Показываем:

Партнёр пока не подключил приём заявок.
15.2 Пользователь ничего не ввёл

Показываем:

Напишите текст запроса.
15.3 Не оставил контакт

Показываем:

Нужен контакт для связи.
15.4 Ошибка отправки

Показываем:

Не удалось отправить запрос. Попробуйте позже.

И бот не падает.

16. Что НЕ делаем на этом этапе

❌ не собираем имя пользователя отдельно
❌ не просим адрес
❌ не просим дату рождения
❌ не строим CRM
❌ не сохраняем историю заявок в сложную базу
❌ не добавляем автоответы партнёра

17. Структура файлов

Логически модуль может выглядеть так:

directory/
 ├ ui/
 │   ├ handlers.py
 │   ├ callbacks.py
 │   ├ request_fsm.py
 ├ render/
 │   ├ request_render.py

Минимум:

обработчик кнопки запроса

FSM состояния

рендер подтверждения

отправка в partner_chat_id

18. Инварианты step 6
1. запрос возможен только у активной карточки
2. request_enabled обязателен
3. partner_chat_id обязателен для отправки
4. flow короткий и линейный
5. пользователь может отменить
6. ошибка не валит систему
7. партнёр получает понятное сообщение
19. Ready check

Шаг считается готовым, если:

1. кнопка "Оставить запрос" открывает flow
2. вводится текст запроса
3. вводится контакт
4. есть подтверждение
5. сообщение уходит партнёру
6. пользователь видит успешный финал
7. при отсутствии chat_id есть честный отказ
8. отмена работает
20. Смысл этого шага

Вот здесь DIRECTORY становится не просто каталогом, а рабочим инструментом:

карточка → запрос → партнёр

Это и есть реальная ценность системы.
