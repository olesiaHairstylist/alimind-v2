MODULE_03_STEP_3 — DIRECTORY CATEGORY MENU
1. Назначение

Этот шаг отвечает за:

переход из раздела "Услуги" в категории DIRECTORY

То есть это первый живой UI-слой DIRECTORY после loader.

2. Место в пользовательском пути

Фиксируем маршрут:

MAIN MENU
→ УСЛУГИ
→ CATEGORY MENU
→ OBJECT LIST
→ CARD

На этом шаге реализуем только:

УСЛУГИ → CATEGORY MENU

И подготовим переход к спискам.

3. Что именно показывает CATEGORY MENU

Пользователь заходит в раздел услуг и видит только категории, в которых есть активные карточки.

Не все возможные категории из справочника мира, а только реальные, живые.

4. Главное правило
КАТЕГОРИЯ БЕЗ АКТИВНЫХ КАРТОЧЕК НЕ ПОКАЗЫВАЕТСЯ

Это важно:

не создаём пустые разделы

не обещаем того, чего нет

меню остаётся чистым

5. Источник данных

CATEGORY MENU строится не вручную в UI и не через захардкоженный список.

Источник:

loader.get_active_categories()

То есть:

loader смотрит реальные active карточки

UI показывает только те категории, которые реально наполнены

6. Формат экрана

Экран должен быть коротким и понятным.

Пример текста:

🛠 Услуги

Выберите категорию.

Без длинных вступлений.

7. Кнопки категорий

Каждая категория = одна кнопка.

Пример baseline:

💇 Красота
🚗 Трансфер
🌍 Переводчики
🔧 Сантехники
🏋️ Спорт / залы
🥊 Тренеры
🏝 Туризм
🏠 Недвижимость
🔨 Ремонт
🚘 Автоуслуги

Но показываются только те, где есть данные.

8. Структура callback

Для стабильности нужен единый формат.

Пример логики:

dir_cat:beauty
dir_cat:transport
dir_cat:translation

То есть:

dir_cat:<category_slug>

Это просто, читаемо и масштабируемо.

9. Что делает нажатие на категорию

На этом шаге кнопка должна уже вести дальше в список объектов категории.

Даже если сам следующий шаг будет описан отдельно, переход должен быть предусмотрен.

Маршрут:

CATEGORY MENU → OBJECT LIST(category)
10. Кнопка назад

Обязательна.

Из CATEGORY MENU пользователь должен вернуться в:

MAIN MENU

Кнопка:

⬅️ Назад

или

⬅️ В меню

Но однообразно по системе.

11. Что НЕ должно быть на этом шаге

❌ не показываем карточки сразу
❌ не мешаем категории и объекты в одном экране
❌ не вставляем контакты
❌ не строим “красивый каталог” раньше времени
❌ не делаем пустые категории для будущего

12. Роль UI-слоя

CATEGORY MENU только:

получает список активных категорий

рендерит текст

строит клавиатуру

передаёт переход дальше

UI не:

фильтрует карточки вручную

не решает бизнес-логику

не хранит данные в коде

13. Нужные части модуля

Логически понадобится:

directory/
 ├ ui/
 │   ├ handlers.py
 │   ├ callbacks.py
 ├ render/
 │   ├ category_menu_render.py

Минимум:

callback для открытия услуг

callback для категории

builder клавиатуры категорий

текст экрана

14. Категории и человекочитаемые названия

Нужно зафиксировать маппинг:

beauty         → Красота
transport      → Трансфер
translation    → Переводчики
home_services  → Сантехники / бытовые услуги
sport          → Спорт / залы
tourism        → Туризм
real_estate    → Недвижимость
health         → Здоровье
daily          → Бытовые сервисы
auto_services  → Автоуслуги

Лучше это держать в одном месте, не размазывать по UI.

15. Инварианты step 3
1. CATEGORY MENU строится только из active карточек
2. пустые категории не показываются
3. UI не хранит список категорий как источник истины
4. каждая категория ведёт в OBJECT LIST
5. есть кнопка BACK
6. экран не перегружен
16. Ready check для step 3

Шаг считается готовым, если:

1. из MAIN MENU открывается раздел "Услуги"
2. показываются только категории с active карточками
3. каждая кнопка категории работает
4. есть возврат назад
5. UI не падает при пустом наборе карточек
6. пустой каталог даёт честный ответ
17. Edge case: карточек нет вообще

Если нет ни одной active карточки, экран должен честно показать что-то вроде:

Сейчас раздел ещё наполняется.

И дать кнопку назад.

Не пустой экран, не traceback.
